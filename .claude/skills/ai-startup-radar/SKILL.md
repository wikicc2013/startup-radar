# 新锐 AI 创业公司研究阵地（ai-startup-radar）

## 一、定位

这不是早报（流），是档案库（库）。每家公司一份持续累积的档案：首次入库做分诊，
A 档做深研，后续批次再出现只做增量标记。产品目标：形成「转型有术视角下的新锐 AI
公司研究阵地」，为公众号、CXO 简报、Agent 厂商图谱供料。

## 二、输入

四类合法输入，处理方式一致：

1. **结构化名单**：xlsx/csv（如 YC batch 表、IT桔子导出）。用 extract-text / pandas 读取。
2. **盘点文章**：md/txt/链接（如 a16z 盘点、公众号文章）。链接必须 web_fetch 全文后再抽取。
3. **混合粘贴**：用户直接贴一段含公司名的文字。
4. **增量 registry**：用户同时提供上期 `startup-radar-registry.md` 时，进入增量模式。

## 三、流水线（六步，顺序执行）

### Step 1 · 抽取
从输入中抽出：公司名、一句话业务、品类、来源出处。**只抽输入里明示的公司**，
不凭记忆补充。抽取结果先展示给用户核对总数（防漏抽/误抽），再进下一步。

### Step 2 · 去重（增量模式）
对照 registry 已有公司名（含常见别名/改名，如 "Atrisa (formerly Refortifai)"）。
已存在 → 不重复建档，在原条目追加 `再现于:{批次编号}`；再现 ≥2 次 → 输出升档提示。
无 registry 时按全量新批次处理，并生成 registry v0。

### Step 3 · 分诊（三档，判据硬编码）

**A 档 · 深研**（预期占比 5-12%），满足任一：
- A1 制造与供应链直击：业务直接服务制造/工业/供应链/汽车汽配/能源/物流仓储/工业分销/
  设备维护/工艺质量/工厂运营；
- A2 组织范式同构：做的事与「AI 重塑组织/岗位/工作流」正面相关——AI 员工、公司大脑、
  人机协同层、流程发现与自动化、AI 原生咨询——即便不碰制造业也入 A 档（方法论对话价值 +
  竞品情报价值）。

**B 档 · 速写卡**（预期占比 8-15%），满足任一且不入 A：
- 企业级 Agent 基础设施（记忆/治理/安全/监控/支付/保险等制度配套）；
- 研发域硬科技（材料/EDA/仿真/测试）；
- 与用户现有雷达体系有交叉信号的（如 Claude Code 生态 → github-radar 交叉）。
B 档在首次入库时只做 300 字速写卡：定位一句话 + 为什么值得记 + 潜在关联场景，不进入默认深研队列；后续专项排水可显式升档深研。

**C 档 · 记名**：其余全部（消费级、纯金融交易、防务武器、泛营销工具等）。
首次入库只在批次原始文件里留名，不进入默认深研队列；后续专项排水可显式升档深研。

分诊结果必须输出漏斗数字（总数 → A → B → C），这是产品纪律。
一个批次 A 档超过 20 家时，向用户提示分两轮深研，不要一次全开。

### Step 4 · 深研（仅 A 档，接力协议）
每家 A 档公司按以下顺序接力，**不在本技能内重写研究逻辑**：
1. `company-research`：公司画像（团队/融资/产品/商业模式/竞争）；
2. `vendor-intel-radar`：客户×场景镜像——但新锐公司客户名单天然稀薄，重点转为
   **早期客户信号**（见第五节）；
3. 产品本身是关键变量时（可试用的软件产品）追加 `product-deep-dive`。
每家深研完成后回写 registry 的「深研状态」为 ✅ 并链接档案文件。
深研节奏由用户控制：默认每次会话深研 2-3 家，不贪多。

### Step 5 · 入库
目录结构（Obsidian vault 内建议路径 `00_CaseInbox/05 AI自动生成/startup-radar/`）：
```
startup-radar/
├── data/companies.json            # 唯一结构化真源
├── startup-radar-registry.md      # 批次与初始分诊的人读索引
├── batches/B{NNN}-{来源}-raw.tsv  # 批次原始名单存档
└── companies/{slug}.md            # A 档单公司档案（B 档速写卡集中放 registry 内）
```

### Step 6 · 输出
每批次产出一份「批次快报」：漏斗数字 + A 档清单及分诊理由 + 3 个最值得写的内容钩子
（供 ai-topic-planner 接力）。深研档案随深研进度逐份交付。

### Step 7 · 统一分类

入库来源的原始品类保存在 `source_category/source_subcategory`；正式的 `category/subcategory`
必须使用 `data/taxonomy.json` 的分类体系 2.0（8 个一级业务域 × 28 个二级产品赛道）。新增公司或
补充研究标签后运行 `node scripts/reclassify.cjs --write`，禁止在同一层混用业务系统名、技术形态和行业名。

## 四、单公司档案模板（companies/{slug}.md）

```
# {公司名}
- 批次来源 / 入库日期 / 再现记录
- 一句话定位（原文 + 中文译注）
- 公司画像（来自 company-research，含融资信息及来源）
- 早期客户信号（见下节，逐条标置信度）
- 转型有术判断：① 对制造业客户意味着什么 ② 国内对标厂商
  ③ 可迁移的干法/该进场景词典的词条 ④ 风险与存疑点
- 信息来源溯源表（无来源的数字一律删除）
```

## 五、早期客户信号（本技能独有的判断层）

新锐公司客户名单稀薄不是缺陷而是信号源：**谁是它的 design partner、灯塔客户是哪家
制造企业，往往是这家公司能不能成的最强信号**。置信度分级沿用案例置信度体系：

| 级别 | 信号形态 |
|---|---|
| 高 | 官方 case study / 客户联合发声 / 官网具名 logo 且客户侧可交叉验证 |
| 中 | 官网 logo 墙（客户侧未验证）/ 权威媒体报道点名客户 |
| 低 | LinkedIn 员工动态推断 / 招聘 JD 暗示 / 社区讨论 |
| 不采信 | 无出处的第三方转述、AI 生成的汇总页 |

## 六、信源策略

- 海外公司：官网 → Crunchbase 公开页 → TechCrunch/The Information/YC 官方目录 →
  LinkedIn/X 创始人动态。
- 国内公司：官网 → 36氪/甲子光年/量子位 → IT桔子**公开页**（仅核对基本信息）。
- **IT桔子铁律**：不爬登录墙/付费墙数据。IT桔子作为输入源的正确姿势是用户人工导出
  清单丢进本技能（与 a16z 文章同一入口），或未来谈正式数据授权后标注引用。
- 融资金额/轮次必须有来源链接，两个信源冲突时并列标注，不裁决。

## 七、失败与边界

- 输入里公司名无法消歧（如 "Modern""Hyper" 这类通用词）→ 用「公司名 + 一句话业务
  关键词 + YC/a16z」组合搜索锚定官网，锚定失败则在档案标注 ⚠️ 未能锚定，不硬写。
- 深研中发现公司已关停/被收购 → 照常建档并标注状态，关停原因本身是研究价值。
- 用户提供的清单超过 300 家 → 建议拆批次入库。
